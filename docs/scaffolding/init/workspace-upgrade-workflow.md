# Workspace Upgrade Workflow

`foundry init upgrade` inspects a workspace and reports whether it matches the current known Foundry init baseline.

The command is currently read-only.

It does not write files.

It does not modify provenance.

It does not overwrite user changes.

## Purpose

Foundry-generated workspaces may need to evolve as `foundry init` improves.

The upgrade workflow exists so a user can answer:

1. Was this workspace generated by `foundry init`?
2. Does it contain Foundry init provenance?
3. Does it contain the current expected baseline files?
4. Is it current?
5. Is an upgrade available?
6. Is manual review required?

## Command forms

Inspect the current directory:

```bash
bun run foundry -- init upgrade
Inspect a specific workspace:

Bash
￼
bun run foundry -- init upgrade myapp
Print the plan as JSON:

Bash
￼
bun run foundry -- init upgrade myapp --json
Use as a CI-style check:

Bash
￼
bun run foundry -- init upgrade myapp --check
Check mode
--check exits successfully only when the workspace status is:

￼
current
--check exits with a non-zero status for:

￼
upgrade-available
not-foundry-workspace
invalid-foundry-workspace
This is intentional.

When using bun run foundry -- ..., Bun may print a message such as:

￼
error: script "foundry" exited with code 1
That is expected when --check intentionally fails.

For scripted negative checks, prefer the wrapper directly:

Bash
￼
if bash tools/scripts/foundry.sh init upgrade tmp-not-foundry --check; then
  echo "Expected upgrade check to fail" >&2
  exit 1
fi
Status values
Status	Meaning	Check mode
current	Workspace matches the known Foundry init baseline.	exits 0
upgrade-available	Workspace is Foundry-generated but missing baseline files.	exits 1
not-foundry-workspace	No Foundry init provenance was detected.	exits 1
invalid-foundry-workspace	Foundry metadata exists, but provenance does not prove foundry init generated it.	exits 1
￼
Detection fields
The command reports:

￼
Foundry directory
Init provenance
Init audit log
Generator manifest
Generated by foundry init
These correspond to:

￼
.foundry/
.foundry/init/provenance.json
.foundry/init/audit.ndjson
config/foundry/generator-manifest.json
generatedBy.command = "foundry init"
Current workspace example
Bash
￼
rm -rf tmp-foundry-init-upgrade-current

bun run foundry -- init tmp-foundry-init-upgrade-current \
  --no-database \
  --yes \
  --no-install

bun run foundry -- init upgrade tmp-foundry-init-upgrade-current
bun run foundry -- init upgrade tmp-foundry-init-upgrade-current --check

rm -rf tmp-foundry-init-upgrade-current
Expected status:

￼
current
Non-Foundry workspace example
Bash
￼
rm -rf tmp-foundry-init-upgrade-not-foundry
mkdir -p tmp-foundry-init-upgrade-not-foundry

bun run foundry -- init upgrade tmp-foundry-init-upgrade-not-foundry

if bash tools/scripts/foundry.sh init upgrade tmp-foundry-init-upgrade-not-foundry --check; then
  echo "Expected --check to fail for non-Foundry workspace" >&2
  exit 1
fi

rm -rf tmp-foundry-init-upgrade-not-foundry
Expected status:

￼
not-foundry-workspace
Upgrade-available workspace
A workspace is upgrade-available when it has valid Foundry init provenance but is missing one or more required baseline files.

Examples of required baseline files include:

￼
.editorconfig
.gitattributes
CONTRIBUTING.md
SECURITY.md
.github/workflows/ci.yml
.github/pull_request_template.md
docs/ai/BOOTSTRAP_PROMPT.md
docs/ai/CURRENT_STATE.md
.foundry/README.md
.foundry/init/provenance.json
.foundry/init/audit.ndjson
The plan lists missing files and planned actions.

Current planned actions are descriptive only.

JSON output
Use JSON output for tooling:

Bash
￼
bun run foundry -- init upgrade myapp --json
The JSON plan includes:

￼
workspaceRoot
status
detected
currentVersion
targetVersion
missingFiles
presentFiles
warnings
actions
Verification
The upgrade workflow is covered by:

Bash
￼
bun run verify:init-upgrade
The full fast verification gate also runs upgrade checks:

Bash
￼
FOUNDRY_SKIP_INIT_VERIFY=1 bun run verify
The full verification gate runs all checks:

Bash
￼
bun run verify
Mutation policy
The current upgrade command is read-only.

A future mutating upgrade command must:

print a plan before writing;

require explicit execution;

avoid overwriting user-edited files without confirmation or backup;

append audit events;

preserve existing provenance;

record upgrade provenance;

be covered by smoke tests;

support a dry-run or check mode;

be safe for CI inspection.

Recommended future command shape
A future mutating command should use an explicit flag such as:

Bash
￼
bun run foundry -- init upgrade myapp --execute
or:

Bash
￼
bun run foundry -- init upgrade myapp --apply
Do not make mutation the default behavior.
