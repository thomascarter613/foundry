Below is the **complete, canonical, governed definition of Document Types** — the *taxonomy* that makes your documentation system **typed**, **machine‑actionable**, **governance‑enforced**, and **knowledge‑graph compatible**.

These are not categories.  
They are **institutional classes** with:

- strict semantics  
- strict required sections  
- strict governance levels  
- strict cross‑link expectations  
- strict directory boundaries  
- strict machine‑readable meaning  

Every document in `docs/` must declare exactly one of these types in its **Governed Header Block**.

Guided Links are embedded so you can explore or refine any type.

---

# 🏛️ **THE SEVEN GOVERNED DOCUMENT TYPES**

These are the **only** allowed document types in the system:

1. **Planning**  
2. **Governance**  
3. **Architecture**  
4. **Lifecycle**  
5. **Standard**  
6. **Platform**  
7. **Onboarding**  

Each type is a **first‑class entity** in the documentation knowledge graph.

---

# 📘 1. **Planning**  
**Purpose:** Define intent, direction, constraints, and conceptual framing.  
**Directory:** `docs/planning/`  
**Governance Level:** Informational or Required  
**Required Sections:**  
- Purpose  
- Context  
- Problem Statement  
- Requirements / Definitions / Maps  
- Constraints  
- Open Questions  
- Change History  

**Machine Meaning:**  
Planning docs shape architecture, scope, and sequencing.

---

# 🏛️ 2. **Governance**  
**Purpose:** Define authority, rules, enforcement, and institutional boundaries.  
**Directory:** `docs/governance/`  
**Governance Level:** Binding  
**Required Sections:**  
- Purpose  
- Authority  
- Scope  
- Rules  
- Enforcement  
- Exceptions  
- Versioning & Lifecycle  
- Change History  

**Machine Meaning:**  
Governance docs constrain everything else — code, architecture, lifecycle, standards.

---

# 🏗️ 3. **Architecture**  
**Purpose:** Define system design, structure, decisions, and constraints.  
**Directory:** `docs/architecture/`  
**Governance Level:** Required or Binding  
**Required Sections:**  
- Purpose  
- Architectural Context  
- Components / Views  
- Constraints  
- Tradeoffs  
- Related ADRs  
- Change History  

**Machine Meaning:**  
Architecture docs drive ADR lineage, diagrams, and structural validation.

---

# 🔄 4. **Lifecycle**  
**Purpose:** Define how work flows through the system.  
**Directory:** `docs/lifecycle/`  
**Governance Level:** Required  
**Required Sections:**  
- Purpose  
- Workflow  
- Roles & Responsibilities  
- Required Artifacts  
- Approval Gates  
- Escalation Paths  
- Change History  

**Machine Meaning:**  
Lifecycle docs drive CI gating, PR rules, and release processes.

---

# 🧪 5. **Standard**  
**Purpose:** Define binding rules for code, APIs, tests, and documentation.  
**Directory:** `docs/standards/`  
**Governance Level:** Binding  
**Required Sections:**  
- Purpose  
- Standard Rules  
- Examples  
- Anti‑Patterns  
- Enforcement  
- Change History  

**Machine Meaning:**  
Standards drive linting, code quality, API shape, and test structure.

---

# ⚙️ 6. **Platform**  
**Purpose:** Define the internal developer platform (IDP) — tooling, CI/CD, observability.  
**Directory:** `docs/platform/`  
**Governance Level:** Required  
**Required Sections:**  
- Purpose  
- Responsibilities  
- Architecture / Components  
- Operational Requirements  
- SLAs / SLOs  
- Change History  

**Machine Meaning:**  
Platform docs drive CI/CD behavior, toolchain governance, and observability rules.

---

# 🧭 7. **Onboarding**  
**Purpose:** Provide human‑readable entrypoints into the system.  
**Directory:** `docs/onboarding/`  
**Governance Level:** Informational  
**Required Sections:**  
- Purpose  
- Audience  
- Overview  
- Key Concepts  
- Navigation Guide  
- Related Documents  
- Change History  

**Machine Meaning:**  
Onboarding docs are the human interface to the living documentation system.

- governable
