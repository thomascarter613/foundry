# 🧩 **1. What Glossary Term Linking *Is***  
Glossary Term Linking is the **mandatory, machine‑enforced requirement** that:

- every glossary term used in any document  
- must link to the canonical glossary entry  
- using a consistent, governed syntax  

Example:

```markdown
The [Monorepo](../planning/glossary.md#monorepo) is the governed structure…
```

This is **binding** and **CI‑enforced**.

# 🧬 **4. Machine‑Actionable Semantics**

Glossary Term Linking is not navigation.  
It is **semantic authority**.

When a glossary term changes:

- all documents using that term are revalidated  
- Quickref updates  
- the knowledge graph updates semantic edges  
- the Docs Agent updates its semantic model  
- drift detection runs  
- ADRs referencing the term are flagged  
- standards referencing the term are flagged  
- architecture docs referencing the term are flagged  

This is how the glossary becomes a **reactive subsystem**.

---

# 🔗 **5. Linking Syntax (Canonical, Binding)**

Glossary terms must be linked using:

```markdown
[Term](../planning/glossary.md#term)
```

Rules:

- lowercase anchor  
- hyphens for spaces  
- no abbreviations  
- no alternate spellings  
- no inline definitions  
- no unlinked glossary terms  

Example:

```markdown
The [Domain](../planning/glossary.md#domain) defines the bounded context…
```

# 🔄 **7. Glossary Term Drift Detection**

Drift detection identifies:

- terms used but not defined  
- terms defined but not used  
- terms used inconsistently  
- terms whose definitions changed but documents didn’t  
- terms whose meaning conflicts with ADRs  
- terms whose meaning conflicts with standards  

This is part of the **semantic drift subsystem**.
