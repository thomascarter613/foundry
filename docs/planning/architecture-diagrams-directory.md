architecture-diagrams-directory.md
Below is the **complete, governed, canonical specification** for the **Architecture Diagrams Directory** — the subsystem that makes your architecture *visual*, *machine‑validated*, *CI‑enforced*, and *knowledge‑graph compatible*.  
This is not “a folder for pictures.”  
It is a **governed architectural visualization platform** inside your monorepo.

Guided Links are embedded so you can expand any subsystem.

---

# 🏛️ **Architecture Diagrams Directory (Canonical, Governed, Machine‑Actionable)**

```
docs/architecture/diagrams/
```

This directory is the **only** place in the entire repository where diagrams may live.  
This rule is **binding** and enforced by:

- **documentation linting engine**  
- **documentation CI rules**  
- **architecture validation engine**  
- **documentation knowledge graph**  

---

# 🧩 **1. Purpose of the Diagrams Directory**

This directory exists to:

- centralize all architectural diagrams  
- enforce deterministic diagram storage  
- prevent diagram drift  
- ensure diagrams are versioned, governed artifacts  
- allow CI to validate diagram references  
- allow the Docs Agent to reason about architecture visually  
- allow the knowledge graph to map visual → textual architecture  

This is the **visual backbone** of your architecture domain.

---

# 🏗️ **2. Allowed Diagram Types (Strict)**

Only the following diagram types are permitted:

- `.png`  
- `.svg`  
- `.drawio`  
- `.mermaid` (embedded in `.md` files)  
- `.plantuml`  

All diagrams must be:

- deterministic  
- version‑controlled  
- reproducible  
- referenced by architecture documents  

---

# 📘 **3. Directory Structure (Canonical)**

```
docs/architecture/diagrams/
  system-context.drawio
  container-diagram.drawio
  component-diagram.drawio
  runtime-views.drawio
  data-flow.drawio
  integration-map.drawio
  <additional governed diagrams>.drawio
```

Each diagram corresponds **exactly** to a governed architecture document.

Example:

- `system-context.md` → `system-context.drawio`  
- `container-diagram.md` → `container-diagram.drawio`  
- `component-diagram.md` → `component-diagram.drawio`  

This 1:1 mapping is enforced by CI.

---

# 🔗 **4. Cross‑Link Rules (Binding)**

Every architecture document must:

- reference its diagram(s) in this directory  
- use relative links  
- include diagram metadata  
- declare diagram dependencies in the knowledge graph  

Example reference:

```markdown
![System Context](./diagrams/system-context.drawio)
```

CI validates:

- link exists  
- file exists  
- file is in the correct directory  
- file type is allowed  

---

# 🧪 **5. Diagram Validation Rules (Machine‑Enforced)**

The **diagram validation subsystem** enforces:

- diagrams must live ONLY in `docs/architecture/diagrams/`  
- no diagrams may appear anywhere else  
- diagrams must match naming conventions  
- diagrams must be referenced by exactly one architecture document  
- diagrams must not be orphaned  
- diagrams must not be duplicated  
- diagrams must not be empty  
- diagrams must not exceed governed size limits  
- diagrams must be deterministic (no random layout)  

These rules are part of the **documentation linting rules**.

---

# 🔄 **6. Reactive Behavior (Living System)**

When a diagram changes:

- the architecture document referencing it must update  
- the knowledge graph updates visual nodes  
- drift detection runs  
- ADRs referencing the diagram are revalidated  
- integration maps update  
- Docs Agent updates architectural reasoning  

This makes diagrams **reactive artifacts**, not static images.

---

# 🧬 **7. Diagram Metadata Schema (Machine‑Readable)**

Each diagram must include a `.md` metadata file:

Example:

```
docs/architecture/diagrams/system-context.meta.md
```

Contents:

```markdown
Diagram: system-context.drawio
Type: System Context
Owned By: Architecture
Last Updated: YYYY-MM-DD
Related Document: ../system-context.md
Related ADRs:
  - ADR 0001: Architecture Principles
  - ADR 0002: Monorepo Structure
```

This metadata powers:

- the knowledge graph  
- diagram validation  
- drift detection  
- Docs Agent reasoning  

---

# ⚙️ **8. Diagram Generation Rules**

Diagrams may be:

- manually created  
- generated by the Docs Agent  
- generated by the architecture generator  
- generated from ADRs  
- generated from the knowledge graph  

But they must always be:

- deterministic  
- reproducible  
- governed  

This is enforced by the **documentation generator architecture**.
