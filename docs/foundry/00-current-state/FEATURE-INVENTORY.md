# Foundry Feature Inventory

## Purpose

This document records the known Foundry feature surface at the start of the controlled reorientation effort.

Feature status values:

- `Complete`
- `Partial`
- `Planned`
- `Experimental`
- `Unknown`
- `Needs Verification`

## Feature Areas

### CLI Foundation

Status: `Partial`

Known elements:

- TypeScript CLI package under `packages/cli`.
- oclif-based command structure.
- Root `foundry` script wrapper.
- Build and typecheck scripts.
- CLI binary configuration.

Needs verification:

- Complete command list.
- Error handling consistency.
- Help text quality.
- Command naming consistency.
- Public CLI usability.

### Repository Initialization

Status: `Partial / Needs Verification`

Known elements:

- `foundry init`.
- Named project mode.
- `--yes`.
- `--no-install`.
- `--no-database`.
- Database provider support.
- Supabase provider family.
- Generated workspace provenance.
- Generated workspace audit metadata.
- Init verification script.

Needs verification:

- Current passing/failing state of `bun run verify:init`.
- Whether every Tier 1 provider still passes.
- Whether generated workspace verification still passes.
- Whether multi-database mode is complete or only specified.
- Whether dry-run behavior is complete.
- Whether collision/path safety behavior is fully covered.

### Database Provider Architecture

Status: `Partial`

Known elements:

- Provider registry concepts.
- PostgreSQL + Drizzle.
- PostgreSQL + Prisma.
- SQLite + Drizzle.
- SQLite + Prisma.
- MongoDB native.
- Supabase SQL.
- Supabase Drizzle.
- Supabase Prisma.
- Supabase client.

Needs verification:

- Actual generated files per provider.
- Provider lifecycle scripts.
- Validation behavior.
- Migration behavior.
- Local service startup behavior.
- Generated `.env.example` completeness.

### Generator System

Status: `Partial`

Known elements:

- Generator manifest.
- scaffdog integration.
- plop integration.
- generator verification scripts.
- generated workspace command surface.

Needs verification:

- Canonical generator registry format.
- Template ownership.
- Template versioning.
- Generator provenance.
- Generator collision policy.
- Generator test fixtures.

### Documentation Governance

Status: `Partial`

Known elements:

- Docs graph commands.
- Docs relationship normalization.
- ADR validation.
- Glossary validation.
- Directory validation.
- Docs readiness report.
- Docs verify strict mode.
- Frontmatter expectations.

Needs verification:

- Which docs are canonical.
- Which docs are stale.
- Which docs are missing frontmatter.
- Which docs fail strict validation.
- Whether repair commands are safe and non-destructive by default.

### Work Packets

Status: `Partial / Planned`

Known elements:

- Work-packet validation scripts.
- Work-packet document directory.
- Work-packet oriented development process.

Needs verification:

- Canonical work-packet schema.
- Required fields.
- Index requirements.
- Relationship to change plans.
- Relationship to issues and commits.
- Relationship to AI/supervised execution.

### Change Plans

Status: `Partial / Planned`

Known elements:

- Change-plan validation scripts.
- Change-plan command surface.

Needs verification:

- Canonical change-plan schema.
- Whether change plans are distinct from work packets.
- Whether change plans should be generated by Foundry.
- Whether change plans should be required before repo evolution.

### Spec-Driven Development

Status: `Experimental / Partial`

Known elements:

- `verify:specs`.
- Spec-kit exploration.
- Desire for specs as first-class workflow artifacts.

Needs verification:

- Current spec schema.
- Spec directory location.
- Relationship among product specs, feature specs, requirements, and work packets.
- Whether Foundry should adopt external spec-kit conventions or implement its own compatible layer.

### Repo Evolution

Status: `Planned / Partial`

Known elements:

- Strong product requirement for `foundry upgrade` or equivalent.
- Desire to evolve existing repositories after initialization.
- Need for non-destructive file writing.
- Need for provenance-aware upgrades.

Needs verification:

- Existing command implementation, if any.
- Upgrade manifest format.
- Diff/preview workflow.
- Safe apply workflow.
- Rollback strategy.
- Compatibility with generated and non-generated repos.

### AI-Native Workflow

Status: `Planned`

Known elements:

- AI usage should be expected but optional.
- Provider-agnostic architecture is required.
- No paid AI subscription should be required.
- Local/self-hosted model compatibility should be possible.
- Repo-native context should support future Charon integration.

Needs verification:

- Current AI-related files.
- Current provider configuration.
- Whether AI belongs in Foundry v1 or AionX later.
- Minimal AI-ready artifacts for Foundry MVP.

### Supervised Execution

Status: `Planned`

Known elements:

- Human remains in control.
- AI proposes work packets.
- System shows diffs.
- User approves before writing.
- Verification runs before commit.
- Push requires explicit approval.

Needs verification:

- Whether this belongs in Foundry CLI MVP.
- Whether this belongs in AionX Workbench later.
- Minimal CLI-only implementation path.
